これは何か

Webアプリではなく、Markdownファイル群 + Git + AIエージェントで構成される「個人用の開発OS」。アイデアを貯め、資料を理解し、要件から実装・リリース・振り返りまでを一貫フローで回し、そこで得た知見を複利で蓄積する。学習・研究・個人開発・創作・キャリア形成を横断して支える「第二の脳」。

特徴は AIに依存しないこと。状態・知識・ルールはすべてMarkdown側に残り、AI（Claude / ChatGPT / Gemini …）は交換可能な実行エンジンとして振る舞う。——そして、このポートフォリオが説明している対象こそ、この仕組みそのものである（システムが自分自身を記録している）。

解決したかった課題 / 制作背景

AIと個人開発を続けるうちに、致命的な非対称に気づいた。知識も状態も「会話の中」に溜まっていくこと。会話は流れて消え、モデルを乗り換えれば失われ、昨日の判断は今日のセッションからは見えない。ツールとしてのAIは強力なのに、その成果が蓄積されない。

「AIを記憶装置として使うのをやめ、AIは実行者に徹させ、記憶は自分が所有するファイルに置く」——この一点を設計の中心に据えて作った。

設計思想（3〜5項目）

  • ファイルが唯一の真実（Single Source of Truth） — 状態はすべてプレーンテキスト。grep でき、Git が履歴を持ち、10年後も開ける。
  • AIはOSではなくエンジン — 運用マニュアルの正典（DEVELOPMENT_OS.md）はAI中立。各AI固有の接着剤は薄いアダプタ（CLAUDE.md / AGENTS.md）に隔離する。
  • UIは投影レイヤー — 表示層（Obsidian・ダッシュボード）はファイルを読むだけ。状態を持たせない。いつでも作り直せる。
  • 規約がツールに勝る — 複雑な仕組みより、一貫した命名・構造・テンプレートで戦う。壊れる部品を減らす。
  • 一情報一箇所 — 同じ情報を複数箇所に持たない（drift防止）。文章＝Markdown、コード＝GitHub、公開サイト＝GitHub Pages。

主要機能

  • 2つのエージェント: Research Agent（資料を要約ではなく構造化理解ノートへ変換）／ Idea Prototyper（曖昧なアイデアを Concept/Feature/設計/Roadmap/Risk/Next へ展開）
  • OSコマンド群（.claude/commands/os-*）: アイデア記録・企画化・プロジェクト生成・タスク分解・バグ管理・開発ログ・リリースチェック・振り返り・知見保存・ポートフォリオ生成 …をスラッシュコマンドで起動
  • プロジェクト管理: 1プロジェクト=1ディレクトリ（requirements/design/tasks/bugs/devlog/handover/retrospective）
  • Knowledge Base: patterns / pitfalls / research を [[slug]] でネットワーク化
  • ポートフォリオ生成: portfolio/<slug>.md を唯一の真実に、スクリプトが決定的にサイトを生成（このページ自身もそれで生成されている）
  • 検証スキル: 単一HTMLアプリの起動確認を必須化する verify-single-html 等

技術構成

  • 状態・知識: Markdown + YAML frontmatter（status/created/updated/tags）
  • バージョン管理: Git（履歴＝「いつ何を決めたか」の台帳）
  • ランタイム: Claude Code（.claude/commands/ = OSのシステムコール）
  • 表示層: Obsidian（グラフ・バックリンク・検索）＋ 単一HTMLダッシュボード
  • ポートフォリオ生成: Node.js（gray-matter + marked）

技術的判断（最重要）

課題: AIとの会話に知識・状態が閉じ込められ、モデルを変えると失われる。
判断: 状態をすべてMarkdownに外部化し、AIを交換可能な実行エンジンにする。運用マニュアルの正典 DEVELOPMENT_OS.md（AI中立）と、各AI固有の薄いアダプタ（CLAUDE.md / AGENTS.md）を分離した。
理由: これによりAIモデルの交代・エディタ変更・UIの作り直しのすべてに耐え、資産（prompts/templates/knowledge/portfolio）がそのまま残る。

課題: チャット履歴に頼ると、新しいセッションで文脈を失い再開できない。
判断: 「Markdown=状態、チャット=一時メモリ」を大原則に、handover.md（上書き＝引き継ぎ「次どこへ」）と devlog.md（追記＝履歴「何をしたか」）を役割分離した。
理由: 新規セッションが handover.md ＋ プロジェクトファイルだけで、過去の会話を一切遡らず完全に再開できる。

課題: 表示UIを作り込むほど保守負担が増え、UIの寿命にデータが縛られる。
判断: 依存方向を「データ層 → ランタイム層 → 表示層」の一方向に固定し、表示層は状態を持たない投影に徹させた。
理由: UIが消えてもデータは無傷。ダッシュボードもObsidianも「いつでも捨てて作り直せる」部品になる。

課題: ポートフォリオがコード・README・公開サイトに分散し、更新のたびに食い違う（drift）。
判断: portfolio/<slug>.md を唯一の真実とし、README・サイトはスクリプトで決定的に生成する（原稿＝人の判断、変換＝機械）。
理由: 一情報一箇所を保て、AIに毎回HTMLを書かせる不確実性も排除できる。

検討した他案

  • 試した案: Notion 等のデータベース型ツールで一元管理 / なぜ捨てたか: ベンダーロックイン・grep不可・オフライン不可・長期保存の不安・AIから直接読み書きしにくい / 最終判断: プレーンテキスト + Git。所有権と可搬性を最優先した。

苦労した点

  • AI非依存の「正典とアダプタ」の切り分け: どこまでを普遍ルール（正典）に、どこからをClaude固有（アダプタ）に置くかの線引き。混ぜると移行できなくなる。
  • セッション境界の設計: 「ファイルが真実」という思想と、実際には会話にメモしてしまう習慣との矛盾を、handover/devlogの分離で解消するまでが試行錯誤だった。

制約・トレードオフ

  • リッチなGUIやリアルタイム性はない（プレーンテキストと引き換え）。
  • 規約前提の運用なので、命名・テンプレート・frontmatterを守る規律が要る。仕組みではなく規約で品質を担保している。

今後追加したい機能

  • knowledge/ 横断検索（実装前に過去の知見を自動参照）
  • 週次レビューの自動生成（全devlogを集計し進捗と停滞を報告）
  • マルチエージェントによる設計・コードレビューの並列化

この作品の位置づけ

自分の制作全体の一貫軸である「学習・創作の体験設計」を、他の作品に適用する側ではなく、開発プロセスそのものに適用した作品。個々のアプリを生み出す母艦であり、「作るために学ぶ／仕組みで問題を解決する」という自分の姿勢が最も濃く出ている。単体のアプリ以上に、どう作り続けるかの思想を形にしたもの。
